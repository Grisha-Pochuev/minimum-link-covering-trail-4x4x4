## Where work was harder than necessary

### 1. Live state was copied into multiple files

`frontier/active_run.json` said smoke was in progress after smoke had already completed, while `START_HERE.md` still said not to start search-24 even though it was running.

Fix: `frontier/active_run.json` is now the only authoritative live-stage record. `START_HERE.md` may contain the run identity and a dated snapshot, but every next chat must defer to `active_run.json` for the current stage.

### 2. The Step-2 launch block was overridden during Step 3

The handoff said to wait for search-23 shard 11 and a strict `20/20` aggregate. Search-24 was launched from the 19 available source shards because all 40 known primary core-escape `62/64` seeds came from available shard 14 and preflight explicitly required at least 19 source artifacts.

This exception was scientifically motivated and recorded in `frontier/active_run.json`, but it should not have been an implicit Step-3 decision.

Permanent rule: a binding Step-2 dependency may be overridden only by a written execution amendment before the trigger, with the exact evidence, lost coverage risk and preflight guard.

### 3. Implementation language drift

The handoff preferred C++20 for enumeration and graph search, but the launched engine is Python multiprocessing. It passed preflight and smoke, but the smoke rate is low enough that this choice may materially limit search depth.

Permanent rule: the Step-2 implementation language is binding. A different language requires a benchmark and a written reason before launch. Do not silently replace a requested C++ search core with Python.

### 4. Permanent-input rule was not fully followed

The workflow materializes search-24 seeds during precheck from retained search-23 Actions artifacts and passes them as a prepared-seeds artifact. This is reproducible for the current run, but it is weaker than committing permanent seed files before smoke.

Required Step-1 cleanup after the full run:

- persist the exact search-24 seed JSONL and manifest, or content hashes plus a lossless reconstruction path;
- record the exact source artifact ids and source shard count;
- ensure the final run archive no longer depends only on expiring Actions storage.

### 5. Too many fragmented source commits

The implementation was published as several sequential file commits. The safe order was good—source first, workflow later, trigger last—but the history is noisier than necessary.

Permanent rule: before writing, build a file manifest and publish one atomic tree commit when the tool supports it. If not, use the smallest practical sequence: implementation bundle, workflow, active-run record, trigger last.
