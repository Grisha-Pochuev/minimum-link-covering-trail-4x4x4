# Retrospective: search-23 Step 3

Date: 2026-07-13

Scope: implementation, smoke validation and full launch for `smart-search-23-core-transplant`.

## Outcome

The technical launch eventually succeeded:

- normal C++20/Python source files were added;
- permanent search-22 seed material was prepared;
- first smoke run `29247958417` stopped at precheck;
- the checker compatibility bug was fixed;
- second smoke run `29248411212` passed precheck, all 20 shards, dual exact checks and aggregate artifact creation;
- full profile was triggered from commit `ac18bf46b23146f4f4a581cbf5af641c746d3171` with `21000` seconds, 20 shards, four workers per shard and `max-parallel=20`;
- `frontier/active_run.json` prevents duplicate launch.

## What worked well

1. The smoke gate protected the expensive run. The checker bug stopped before any full 5h50m search was launched.
2. Full was not launched after merely seeing a green precheck. The complete smoke aggregate and shard artifacts were checked first.
3. The implementation used readable normal modules rather than another runtime-concatenated fragment system.
4. Exact verification remained separate from the C++ heuristic engine.
5. The selected Step-2 hypothesis was preserved instead of being replaced during implementation.

## What went wrong

### 1. Step 3 was split across user prompts

The intended contract was one user prompt covering implementation, smoke and full launch. In practice the first smoke failed, the user saw a ChatGPT safety warning and interrupted for status, then later supplied the successful smoke URL and requested the full launch.

The root process problem was not the interruption itself. The workflow had separate smoke and full triggers and no automatic dependency from successful smoke aggregate to full. Therefore the chat had to observe one run and create a second trigger. This makes Step 3 fragile whenever the conversation is interrupted.

### 2. Local dry-run did not reproduce the exact GitHub precheck

The first smoke failed because the two new checkers required a `links` field while compact seed records represented the trail with 23 vertices and did not always contain that redundant field.

The launch document required both exact checkers over all 12 required seeds, but the local test did not execute exactly the same commands and files as GitHub precheck. The dry-run therefore gave false confidence.

Fix: every numbered search must provide one versioned preflight command used unchanged both locally and in GitHub precheck. Seed schema normalization must happen before both checkers.

### 3. The workflow modified the repository during precheck

The precheck downloaded the expiring search-22 artifact and committed permanent seed material to `main` from inside GitHub Actions. The run itself continued from its original launch commit while the branch advanced.

This worked, but it creates unnecessary state races, makes provenance harder to read and produces an extra commit during validation.

Fix: all permanent seeds and manifests must be committed before the smoke trigger. A search workflow should be read-only with respect to repository contents. It may upload artifacts, but should not commit source or seed preparation back to `main`.

### 4. Too many connector write calls and commits

The implementation was uploaded through many individual `create_file` operations. This produced a long chain of small commits and increased the number of tool calls exposed to transient connector or safety-layer failures.

Fix: prepare the complete coherent change locally, then publish it as one atomic Git tree commit when the connector supports blobs/tree/commit/ref. If that is unavailable, use the smallest practical number of coherent commits.

### 5. Aggregate attempted to run after failed precheck

In the first smoke, the shard matrix was skipped but the aggregate job still started and failed because no shard outputs existed. This added noise without new information.

Fix: the scientific aggregate runs only after all required shard jobs succeed. A separate lightweight failure-report job may use `if: always()`, but it must not pretend to aggregate missing scientific outputs.

### 6. Safety warning was confused with CI failure

One GitHub connector write was blocked by the ChatGPT safety layer because it could not classify the request. A smaller explicit retry succeeded. This was separate from the first smoke failure, which was an ordinary checker/schema bug.

There is no evidence that ChatGPT forbids the assistant from launching GitHub Actions. Self-launching does require more write operations, so it encounters more opportunities for a false-positive tool block. The correct response is to verify repository state, reduce or clarify the write, and retry only the non-trigger operation. Never repeat a launch trigger unless repository and Actions state prove that no run was created.

## New Step-3 contract

The four project steps remain unchanged. Step 3 becomes a single-prompt transaction:

1. implement the saved Step-2 handoff;
2. run the exact shared preflight locally;
3. publish one coherent implementation commit;
4. trigger an automatic workflow profile;
5. run smoke matrix;
6. require smoke aggregate, exact checks and every expected artifact;
7. automatically start the full matrix inside the same numbered workflow after the smoke gate succeeds;
8. confirm the full profile and all 20 jobs;
9. record the active run and stop without asking the user to trigger anything.

Preferred workflow shape for future searches:

```text
precheck
  -> smoke[20]
  -> smoke-aggregate
  -> full[20]             only if smoke-aggregate succeeds
  -> full-aggregate
```

The workflow may still expose manual `smoke` and `full` profiles for debugging, but normal Step 3 uses an `auto` profile or one narrow `auto.trigger` file.

## Rules carried forward

- Keep exactly four user-facing steps.
- Step 3 must not end merely because smoke was launched.
- Do not require the user to return a smoke URL or send a second launch command.
- If smoke fails, diagnose, fix and restart the automatic chain within the same Step-3 work whenever permissions allow.
- Do not alter the primary mathematical hypothesis during Step 3.
- Do not modify the already launched search-23 workflow commit; apply the automatic-chain architecture starting with the next numbered search.
