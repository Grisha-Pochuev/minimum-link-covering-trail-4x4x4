## Interpretation

This run was computationally successful but mathematically incomplete.

It used the GitHub Actions setup well: 16 parallel jobs, 4 workers per job, nearly the full safe wall-time window, and successful artifact upload from every shard. It did not find a 22-link solution and did not come very close to full coverage: the best result remained 10 points short.

The main limitation is the search model. It searches only integer vertices in [-3, 7]^3. The checker also currently converts input coordinates to int, so it is not ready for rational vertices. For this problem family, rational vertices such as half-integers may be important. Therefore the next run should not merely repeat the same integer search with more time.

## Recommendation for the next run

The next run should be a smarter third run rather than a brute-force repeat.

It should keep the strong GitHub Actions layout, but improve the search space and scoring:

- keep 16 parallel jobs and 4 workers per job, unless we intentionally choose 8 jobs for a 32-vCPU run;
- keep about 5h40m per job, below the 6-hour limit;
- preserve all artifacts and produce one combined summary artifact at the end;
- add warm-start search from the best 54/64 and 53/64 trails;
- add a mode that targets the recurring missing points directly;
- add rational vertices at least with denominator 2;
- rewrite the checker using exact Fraction arithmetic instead of int conversion;
- add a loss-budget filter for 21-link and 22-link searches;
- build a catalog of useful maximal segments and collapse duplicate segments with the same covered grid-point mask;
- split workers by strategy, for example warm-start 22, fractional 22, strict 21, segment-catalog 22.

Bottom line: the run did not solve the 22-link case, but it gave a useful negative result and a map of where the current search gets stuck. The next launch should use the 54/64 candidates and repeated missing-point frequencies as input, not start from scratch.
