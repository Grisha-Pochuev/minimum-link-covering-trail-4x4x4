# Следующий этап: параллельный поиск и разбор дефекта

Текущее состояние репозитория такое: у нас есть проверяемая 23-звенная ломаная и есть поиск 22-звенной ломаной в целочисленном окне. Это хороший первый слой, но старый ночной запуск использует один процесс Python внутри каждого задания GitHub Actions. У обычного `ubuntu-latest` есть 4 vCPU, поэтому один процесс оставляет заметную часть машины без работы.

Этот этап добавляет две вещи.

Первая вещь: `scripts/search_22_parallel.py`.

Он не заменяет старую геометрию и не меняет смысл поиска. Он берет уже существующий `search_22_integer_box.py` и запускает несколько независимых работников внутри одного задания. По умолчанию работников 4. То есть один осколок начинает использовать все 4 vCPU, а не одно ядро.

Вторая вещь: `scripts/analyze_22_results.py`.

Этот скрипт нужен после скачивания артефактов. Он собирает все JSON-файлы, смотрит лучшие покрытия и отдельно считает повторяющиеся наборы недостающих точек. Это важно, потому что если разные осколки снова и снова не добирают одни и те же точки, значит мы видим не случайный провал, а устойчивый дефект формы поиска.

## Что это дает математически

Пока это еще не доказательство, а усиление вычислительной разведки.

Но оно полезно по двум причинам.

- Если 22-звенная ломаная существует в этом классе поиска, шанс найти ее становится выше, потому что ночной запуск использует больше реальной мощности.
- Если 22-звенная ломаная не находится, мы получаем не просто «ничего не нашли», а карту повторяющихся дыр. Из нее уже можно делать следующую лемму: какие точки систематически конфликтуют между собой и почему один недостающий слой нельзя закрыть без потери другого.

Для чайника это можно понимать так: раньше мы просто много раз пробовали собрать почти идеальную цепочку. Теперь мы еще и собираем статистику, где именно она ломается. Это переход от «искать наугад» к «понять, где застревает задача».

## Как запускать

После добавления файлов откройте:

```text
Actions -> overnight-22-parallel-search -> Run workflow
```

Оставьте значения по умолчанию:

```text
seconds = 20400
box_min = -3
box_max = 7
top_k = 32
workers = 4
```

После завершения скачайте артефакты `parallel-22-shard-*`.

Если хотите разобрать все скачанные JSON локально или в следующем запуске, положите их в папку `results` и выполните:

```bash
python scripts/analyze_22_results.py results --out results/combined_22_summary.md
```

## Как читать результат

Полная удача выглядит так:

```text
covered_count = 64
missing = []
links <= 22
```

Почти удача тоже полезна:

```text
covered_count = 63
missing = [[x, y, z]]
```

или

```text
covered_count = 62
missing = [[x1, y1, z1], [x2, y2, z2]]
```

Главное смотреть не только на лучший один файл, а на повторяющиеся `missing`. Повторяющийся недобор — это подсказка для следующей математической редукции.
